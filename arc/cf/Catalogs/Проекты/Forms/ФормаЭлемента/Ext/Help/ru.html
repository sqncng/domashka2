Проект

Проекты в СППР соответствуют разрабатываемым конфигурациям.
Большинство объектов информационной базы в СППР относится к тому или иному проекту. По проектам ведется разграничение доступа пользователей к информации, хранящейся в информационной базе.
Каждый проект включает основные настройки, касающиеся работы с информационной базой и относящиеся именно к данному проекту (правила обработки ошибок, настройки работы с метаданными, необходимость рассылки уведомлений и т.п.)

Для каждого проекта определяется, нужно ли вести список обработчиков обновления информационной базы. Эта функциональность доступна только для конфигураций, которые разрабатываются на основе Библиотеки стандартных подсистем.

Если обработчики обновления для проекта ведутся, то при загрузке обработчиков обновления в проекте автоматически обновляется таблица библиотек, которые поставляют обработчики обновления.

Для проектов в которых предусмотрена локализация в таблицу библиотек необходимо добавить библиотеку локализации и модуль локализации с соотвествующей отметкой (модуль локализации может быть только один). При генерации кода процедур обработки обновления для этого модуля будет формироваться отдельная процедура содержащая обработчики обновления локализованных объектов.

Как используются поля Имя конфигурации, Наименование конфигурации, Краткое наименование конфигурации?

Поле Имя конфигурации содержит имя конфигурации (идентификатор) так, как оно задано в метаданных конфигурации.

В поле Наименование конфигурации указывается, как правило, официальное наименование конфигурации.

Для чего используются настройки проекта на закладке Работа с метаданными?

При разработке проекта метаданные и обработчики обновления разрабатываемой конфигурации могут загружаться непосредственно в СППР в справочники Объекты метаданных и Обработчики обновления информационной базы. Для возможности загрузки метаданных и обработчиков обновления требуется выполнить ряд настроек на данной закладке. Непосредственно загрузка метаданных и обработчиков обновления производится при помощи обработки Загрузка метаданных, либо при помощи регламентного задания.

Для чего используется табличная часть Генерация описаний обработчиков обновления на закладке Работа с метаданными?

Для проектов, разрабатываемых на основе библиотек стандартных подсистем, предоставляется возможность вести описания по обработчикам обновления информационной базы. Табличная часть Генерация описаний обработчиков обновления позволяет хранить сведения о библиотеках и подсистемах, включенных в проект.

В таблице хранится информация о:

  • библиотеках и подсистемах проекта;
  • общих модулях, которые содержат описания обработчиков обновления;
  • режимах выполнения отложенных обработчиков обновления, включенных в библиотеки и подсистемы проекта;
  • версиях библиотек, начиная с которых осуществляется поддержка параллельного режима выполнении отложенных обработчиков;
  • модуле локализации, который содержит обработчики обновления локализованных объектов (в конфигурации может быть только один модуль локализации)

Данные в таблицу можно загрузить из разрабатываемой конфигурации при помощи обработки Загрузка метаданных.

Информация, хранимая в табличной части используется для построения очереди выполнения отложенных обработчиков обновления, включенных в подсистемы и библиотеки, поддерживающих параллельный режим выполнения отложенных обработчиков, а также для генерации кода процедуры ПриДобавленииОбработчиковОбновления подсистем проекта и процедуры ПриФормированииОчередейОтложенныхОбработчиков библиотек стандартных подсистем.

Какие настройки необходимо выполнить для загрузки метаданных и обработчиков обновления в СППР?

Загрузка метаданных возможна из проекта, ведущегося с использованием GIT, либо из хранилища конфигурации. Для загрузки обработчиков обновления необходима информационная база конфигурации, разрабатываемая на основе библиотек стандартных подсистем. Базу рекомендуется разместить на сетевом ресурсе и использовать только для целей загрузки метаданных в СППР. При этом необходимо учесть, что при работе с системой проектирования прикладных решений в клиент-серверном варианте загрузка метаданных производится на сервере. Для этого на сервере запускается конфигуратор в пакетном режиме, конфигурация выгружается в файлы, из которых выполняется загрузка метаданных в СППР.

Чтобы обеспечить возможность такой работы, на сервере должен быть установлен конфигуратор соответствующей версии (при необходимости - несколько версий), а пользователь, от имени которого работает сервер 1С:Предприятия, обслуживающий СППР, должен иметь права на работу с информационной базой, используемой для выгрузки метаданных.

В поле Ветка разработки необходимо указать ветку, в которой разрабатывается проект.
В зависимости от настроек ветки, доступны поля для доступа к GIT-репозиторию, либо к хранилищу конфигурации, которые необходимо заполнить.

При использовании хранилища конфигурации для того, чтобы в СППР загружались актуальные метаданные, необходимо актуализировать метаданные в конфигурации, используемой для загрузки, Рекомендуется обновлять конфигурацию из хранилища, в котором ведется разработка проекта. Для возможности автоматического обновления из хранилища требуется указать некоторые настройки:

  • Имя пользователя - имя пользователя хранилища
  • Пароль - пароль пользователя хранилища.
  • Версия платформы -номер версии платформы используемой для работы с хранилищем.
  • Всегда обновлять конфигурацию ИБ при загрузке метаданных - если флаг установлен, перед обновлением метаданных будет выполняться автоматическое обновление ИБ из хранилища. Это позволяет контролировать актуальность загружаемых метаданных. Пользователь, обладающий полными правами, или администратор проекта может при загрузке метаданных отключить обновление ИБ из хранилища.

При разработке технических проектов измененные метаданные попадают в основное хранилище, как правило, когда технический проект разработан. Однако до этого момента также может потребоваться оперировать метаданными, создаваемыми и измененными в рамках технических проектов. для этого в СППР имеется возможность загрузки метаданных из хранилищ технических проектов. Для загрузки метаданных из хранилищ технических проектов необходимо указать ряд настроек:

  • Адрес - адрес расположения хранилищ технических проектов - каталог на сетевом диске, или адрес сервера хранилищ, например tcp://Server:2542/.Адрес хранилища конкретного технического проекта указывается в ветке технического проекта.
  • Версия платформы - номер версии платформы, используемой для работы с хранилищами технических проектов.

Как настроить загрузку изменений в ветках?

Для автоматической загрузки изменений в ветках нужно установить флажок Загружать изменения метаданных в ветках и указать расписание загрузки.

По указанному расписанию будет выполняться загрузка изменений. С этими изменениями можно работать в соответствующем списке.

Какие настройки необходимо выполнить для возможности работы с ошибками по проекту?

Для работы с ошибками необходимо задать ряд настроек, расположенных на закладке Работа с ошибками:

  • Порядок воспроизведения и исправления ошибок - в таблице указывается список версий проекта, по которым имеет смысл регистрировать ошибки. Таблица должна актуализироваться при появлении новых версий, прекращении поддержки старых версий. При регистрации ошибок по проекту для указания версий, в которых воспроизводится ошибка, доступны только версии, указанные в данной таблице. Если версия не поддерживается и ошибки в ней не рассматриваются, ее не следует включать в таблицу, при этом у пользователей не будет возможности зарегистрировать ошибки данной версии. Для каждой версии может быть определена настройка - в какой версии следует исправлять ошибку, в зависимости от срочности ее исправления. Также можно выполнить отдельную настройку. для тех ошибок, информация о которых получена из внешних источников. Таким образом, ответственный за планирование исправления ошибок может регулировать процесс исправления ошибок. При этом пользователь, не обладающий правом планирования исправления ошибок, не может самостоятельно изменить версию исправления ошибки, а указывает лишь версию, в которой ошибка обнаружена и срочность исправления. В зависимости от настроек, указанных в проекте, система автоматически определяет версию, в которой требуется исправить ошибку.
  • Контролировать обязательность публикации ошибок из внешних источников - при установке данного флага система помогает обратить внимание пользователя на необходимость публикации ошибок. При этом ошибки, информация о которых получена из внешних источников, предлагается считать требующими публикации, т.к. информация о таких ошибках может быть особо важна пользователям. Информация о необходимости публикации отображается в форме ошибки.
  • Указывать в ошибках раздел проекта - определяет необходимость указания в ошибках раздела проекта. Указание раздела проекта позволяет отбирать ошибки по тому или иному разделу. Кроме того, при регистрации ошибок с указанием раздела проекта ошибки автоматически направляются ответственному за раздел проекта.
  • Указывать в ошибках актуальных версий проекта, кто привнес - определяет необходимость указания в ошибках сотрудника, который привнес ошибку, при исправлении. При выборе этой настройки, поле "Кто привнес" на закладке "Исправление" формы ошибки становится обязательным к заполнению для актуальных версий проекта. Актуальная версия проекта - это та, которая представлена в таблице "Порядок воспроизведения и исправления ошибок" в справочнике Проекты и у которой дата окончания поддержки позже, чем текущая дата.
  • Расположение баз, прикладываемых к ошибкам - указывается каталог общего ресурса, используемый для размещения информационных баз, необходимых для воспроизведения ошибки. При добавлении баз к ошибкам автоматически предлагается разместить базы в указанном каталоге.
  • Ответственный за объекты метаданных - указывается пользователь, ответственный за работу с метаданными по проекту в целом. При регистрации ошибок с использованием веб-сервиса регистрации ошибок, например, при помощи АПК (автоматизированной проверки конфигураций) ошибки, выявленные в конкретных объектах метаданных, направляются ответственному за объект метаданных. Если же для объекта метаданных ответственный не указан, ошибки направляются ответственному за метаданные по проекту в целом.
  • Направление ошибок на проверку - позволяет автоматически направлять ошибку при ее исправлении пользователю. в зависимости от данной настройки. Ошибка может направляться для проверки тому, кто ее зарегистрировал, либо исправившему.Также она может быть направлена ответственному за раздел проекта, либо конкретному пользователю, указанному в настройке.
  • Закрытие ошибок - позволяет организовать процесс обработки ошибок с их закрытием, когда для окончательного завершения работы с ошибкой требуется ее закрыть, т.е. перевести ее в статус Закрыта. При этом для ошибок, которые не находятся в данном статусе, работа с ошибками считается незавершенной. Использование закрытия ошибок позволяет организовать дополнительный контроль за обработкой ошибок. При выполнении настройки можно определить пользователя, которому будут направляться ошибки на закрытие. При этом пользователь должен обладать правом закрытия ошибок. Пользователи, не обладающие таким правом, не могут самостоятельно закрывать ошибки и должны направлять ошибки на закрытие пользователю, ответственному за их закрытие.
  • При исправлении ошибки автоматически отмечать исправление в ветках - если флаг установлен, при исправлении ошибки автоматически будут установлены отметки об исправлении ошибки во всех ветках, указанных в ошибке (при указании варианта исправления ошибки "В разных ветках".

Для чего требуется настройка использования проектов -библиотек?

Информация об использовании проектов-библиотек в системе помогает планировать и контролировать выпуск проектов с учетом сроков выпуска библиотек. Кроме того, при ведении ошибок система автоматически определяет наличие ошибок библиотеки в проекте, включающем эту библиотеку.

Для каждой библиотеки, используемой в проекте, можно указать, какие разделы этой библиотек включены в проект, т.к. некоторые библиотеки могут внедряться частично, например БСП (библиотека стандартных подсистем). При определении наличия ошибок в проекте система выполняет анализ с учетом используемых разделов библиотеки (если раздел не используется в проекте, то ошибки по данному разделу отсутствуют в проекте).

Для указанных в проекте библиотек можно включить ведение справки в самой библиотеке. При таком подходе в проекте не нужно отдельно вести описание справки для библиотечных объектов, исключая тем самым дублирование. При этом в элементах справки библиотеки появляется возможность указать, для каких проектов предназначена справка.

Для чего служат настройки уведомлений?

По ряду объектов СППР система может формировать уведомления, направляемые в виде сообщений по электронной почте пользователям. Такие уведомления могут формироваться по сообщениям, ошибкам, задачам и техническим проектам. Каждый пользователь может персонально для себя решить, хочет ли он получать уведомления и определить собственные настройки в форме Персональные настройки.

Уведомления по сообщениям формируются при направлении новых сообщений пользователю. Уведомления по задачам и ошибкам формируются при направлении задачи или ошибки исполнителю, а также при выполнении (исправлении) задачи/ошибки. По техническим проектам уведомления формируются по факту прохождения контрольных точек.

Уведомления формируются в том случае, если в настройках проекта установлен соответствующий флаг. Рассылка уведомлений производится при помощи регламентного задания, расписание которого может настроить администратор.

Для чего служит настройка Использовать в требованиях раздел проекта?

При установленном флаге в требованиях становится доступно указание раздела проекта. Отнесение требований к разделам позволяет группировать требования по разделам.

Для чего служит настройка Согласующий трудозатраты по задачам?

При помощи данной настройки можно определить, какому пользователю по умолчанию будут направляться задачи для согласования трудозатрат в случае, если пользователь, работающей с задачей, не имеет права на согласование трудозатрат.

Для чего служат настройки тестирования?

Флаг Запускать автоматическое тестирование позволяет влючить запуск тестов в системе непрерывного тестирования согласно расписанию тестирования заданному в СППР.

Флаг Хранить сценарии работы пользователей в СППР определяет, что разработка сценариев тестирования может происходить в справочнике Сценарии работы пользователей

Флаг Хранить сценарии работы пользователей в репозитории определяет, что разработка сценариев тестирования может происходить с помощью интеграции с git сервером.

Флаг Загружать результаты выполнения тестов включает механизм загрузки результатов выполнения тестов из системы непрерывного тестирования с помощью регламентного задания.

Флаг Проверять наличие запроса на помещение при смене статуса ветки включает механизм контроля наличия или отсутствия связанных запросов на помещение ветки в системе хранения исходного кода.

Как настроить приоритеты для проекта?

Настройку приоритетов для проекта можно выполнить на закладке "Прочие настройки".
Настройка доступна, если в системе включено использование приоритетов.
Для выполнения настройки необходимо включить использование приоритетов для проекта, и нажать на гиперссылку, в которой отображается текущее состояние настройки.
Настройка приоритетов считается корректной, если задано как минимум два приоритета и указан приоритет по умолчанию.
