Копирование, сравнение, обновление объектов

Обработка предназначна для копирования, сравнения и обновления объектов функциональной модели (функций, профилей пользователей, видов доступа). Использование этой возможности позволяет в одном проекте (приемнике) использовать функциональную модель другого проекта (источника). Например, некоторая библиотека используется в прикладном решении и для описания функциональности решения в части, относящейся к библиотеке, можно использовать готовую функциональную модель библиотеки, а не описывать ее заново.

Если для проекта ранее не устанавливалась связь с объектами другого проекта, используемого в качестве источника, автоматически включается режим копирования объектов, при котором на основании объектов проекта-источника в проекте-приемнике создаются аналогичные объекты. Пользователь при этом может указать правило использования объекта. Если же объекты ранее были скопированы, то включается режим сравнения, который показывает установленное соответствие между объектами проекта-приемника и источника.

Для чего и как нужно указывать правило использования объектов источника?

Правило использования определяет, каким образом будет использован объект источник.

  • Без изменений - объект заимствуется "как есть", изменений при этом не предполагается. Например, конкретная функция системы тспользуется в том виде, в котором она реализована в источнике.
  • С изменениями - в объект могут вноситься изменения. Например, в рамках существующей функции может потребоваться дополнительная информация-. потребуется дополнительный вход и функция будет изменена.
  • Не используется - объект не задействуется в проекте-приемнике. Например, использование конкретной функции не требуется.

Если для объектов источника и приемника ранее не было проставлено соответствие и не указаны правила использования, то по умолчанию предлагается создание (копирвоание) объектов в проекте-приемнике с правилом использования Без изменений. В дальнейщем правило использования может быть изменено. Также можно изменить его непосредственно перед копированием объектов, в том числе с использованием групповых команд установки правила.

Как можно понять, в чем заключается различие в объектах, если объекты были изменены?

Если объект-источник изменился (например у функции появились новые входящие данные), при использовании функциональной модели источника необходимо понять, в чем заключаются отличия, необходимо ли отразить эти изменения в проекте-приемнике.

Для анализа различий объектов можно воспользоваться контекстными командами для интересующих объектов. При этом сравнить можно как объекты источника и приемника между собой, так и изменения в источники (или приемнике), произошедшие с момента последнего обновления данных приемника на основании источника.

Для удобства можно отобрать только те объекты источника, которые были изменены. Таким образом, например, при обновлении функциональной модели источника (при получении новой версии) можно проанализировать, какие изменения следует перенести в объекты приемника. Отобрать измененные объекты можно при помощи фильтра "Фильтр по объектам источника".

Почему могут возникать сообщения о невозможности обновления, или копирования объектов?

Объекты функциональной модели тесно связаны между собой. Например, функции могут иметь дочерние функции, одни функции могут являться потребителями результата других функций. В качестве исполнителей функций выступают профили пользователей, а профили пользователей включают в себя виды доступа.

При этом, если объект источника используется без изменений, то он используется целиком (если не использовать дочернюю функции для родительской функции, или не использовать профиль, являющийся исполнителем функции, то объект будет уже измененным. Поэтому система контролирует взаимосвязи объектов с учетом указанных правил использования, выдавая в случае необходимости сообщения пользователю.
